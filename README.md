# tlb-simulation
This project explores the performance of different Translation Lookaside Buffer (TLB) replacement algorithms under a variety of simulated memory access scenarios. The goal is to understand how algorithm design and workload characteristics impact TLB efficiency.

# how to run 
1. Open the Python file
2. Run the program (e.g., press ▶ Run or use: python filename.py)
3. View the TLB simulation results printed in the console
Note: The program uses randomness, so results may vary slightly each run.

# write-up
In this project, I simulated a Translation Lookaside Buffer (TLB) to evaluate how different replacement algorithms perform under varying memory access patterns. The goal was to understand how locality and TLB size impact performance. To model realistic process behavior, I generated memory access patterns with locality. Each process accesses a working set of memory addresses, and with a high probability (0.9), accesses stay within this working set. I was able to simulate different runtime scenarios: strong locality resulted in repeated access to the same addresses, while weaker locality introduced more randomness and reduced predictability. I implemented a pseudo TLB with variable sizes (4, 8, 16, 32 entries), reflecting the real-world idea that accessing memory through the TLB is significantly faster than going to main memory. Larger TLB sizes allow more addresses to be stored, which improves performance. I implemented three different TLB replacement algorithms: LRU (Least Recently Used), which removes the least recently accessed entry; FIFO (First In First Out), which removes the oldest entry; and Random, which removes a random entry. Each algorithm makes different decisions about which entry to evict when the TLB is full, directly impacting performance. I used hit rate, miss rate, and average access time to determine how efficiently each algorithm uses the TLB. LRU consistently performed the best because it takes advantage of locality more effectively. FIFO did not perform as well because it does not consider how frequently or recently an address is used, which can result in useful entries being removed too early. Random performed the worst because it evicts entries without any strategy. As the TLB size increased from 4 to 32, hit rates increased and average access times decreased. When locality was strong, all algorithms performed significantly better. Overall, this project demonstrates that both TLB size and the choice of replacement algorithm significantly affect system performance.
